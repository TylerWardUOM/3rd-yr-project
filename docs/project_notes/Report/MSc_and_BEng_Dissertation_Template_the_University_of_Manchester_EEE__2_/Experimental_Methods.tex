This chapter describes the experimental procedures used to evaluate the communication performance, closed-loop timing behaviour, simulation fidelity, and safety behaviour of the proposed haptic rendering system.

The evaluation was designed to assess whether the implemented architecture satisfies the real-time requirements necessary for stable haptic interaction, with particular focus on communication reliability, timing determinism, and force-feedback safety.

\subsection{Communication Performance Tests}

To evaluate serial communication performance independently of the main application, a dedicated command-line test utility was developed. The tool provided two operating modes: a packet rate and integrity mode for analysing streamed device packets, and a ping--echo mode for measuring round-trip latency.

In both modes, incoming serial data were processed using a buffered parser with explicit packet header search, checksum validation, and byte-wise re-synchronisation to ensure robustness to fragmented or corrupted input. Test outputs were written to CSV files for offline statistical analysis, including effective receive rate, inter-arrival jitter, packet drops, out-of-order events, and RTT percentile metrics.

The communication tests were conducted across a range of target transmission rates from 100~Hz to 2000~Hz in order to assess the effect of increasing packet frequency on communication reliability and timing performance.

\subsubsection{Packet Integrity and Rate Tests}

Packet integrity and communication timing were evaluated using the dedicated rate-test mode of the communication test utility.

For each target transmission frequency, the firmware was configured to stream fixed-format device state packets at a constant rate. Tests were performed at nominal frequencies of 100~Hz, 250~Hz, 500~Hz, 1000~Hz, 1250~Hz, and 2000~Hz.

For each test condition, the host continuously received and parsed incoming packets over the USB virtual COM interface. The following performance metrics were recorded:

\begin{itemize}
	\item valid packet count
	\item dropped packets inferred from sequence gaps
	\item out-of-order packets
	\item host-observed inter-arrival time
	\item effective receive rate
	\item inter-arrival jitter
\end{itemize}

All results were exported to CSV format for offline statistical analysis and visualisation.

\subsubsection{Round-Trip Latency Test}

Round-trip latency was evaluated using a ping--echo communication test.

In this test, the host transmitted timestamped ping packets to the embedded controller, which immediately returned an echo packet containing the original sequence number and timestamp.

Tests were performed using transmission intervals of 1~ms, 2~ms, and 10~ms.

For each test, the following metrics were computed:

\begin{itemize}
	\item mean RTT
	\item median RTT
	\item 95th percentile RTT
	\item 99th percentile RTT
	\item minimum and maximum RTT
	\item timeout count
\end{itemize}

These results were used to characterise communication latency and timing consistency under different transmission loads.

\subsubsection{End-to-End Host-Side Timing Test}

In addition to the standalone serial communication tests, end-to-end timing of the host-side closed-loop haptic pipeline was evaluated using a dedicated runtime logging framework integrated into the main application.

During normal operation of the haptic system, the device communication thread published structured timing and state log messages to a dedicated logging subsystem. To avoid perturbing the real-time control path, log messages were transferred asynchronously to a separate logging thread and buffered in memory during execution. After the application was stopped, the captured logs were exported to CSV files for offline analysis in MATLAB.

Two primary log files were generated. The first, \texttt{device\_timing.csv}, contained matched state-command timing records for closed-loop control cycles, including host-side timestamps associated with packet parsing, tool-state publication, wrench consumption, and torque command transmission. The second, \texttt{device\_state\_log.csv}, contained a record of every valid parsed incoming state packet, including host receive timestamps, packet sequence identifiers, MCU timestamps, and measured joint angles.

From these logs, the following metrics were evaluated:

\begin{itemize}
	\item host-side end-to-end latency between state-packet parsing and torque command transmission
	\item intermediate host pipeline delays between communication, haptic processing, and transmission stages
	\item matched state-command sequence consistency
	\item parsed state-packet inter-arrival timing on the host
	\item sequence gap statistics for identifying skipped packets
	\item estimated parsed packet rate and MCU-side timing trends
\end{itemize}

Because the host and microcontroller clocks were not synchronised, the analysis was restricted to host-side timing differences and trend-based interpretation of MCU timestamps. Absolute one-way latency between host and device could therefore not be inferred directly.


\subsection{Simulation Validation}

Simulation validation tests were designed to assess both the correctness of the haptic rendering model in isolation and the stability of the integrated system during interaction.

Two complementary forms of evaluation were performed. First, the behaviour of the virtual coupling and contact model was assessed by observing the computed force output of the simulation under controlled interaction conditions. Second, the full system, including the physical device, communication pipeline, and actuator feedback, was evaluated during sustained user interaction.

\subsubsection{Virtual Wall Test}

A virtual wall test was performed to validate the basic unilateral contact behaviour of the haptic rendering engine. In this test, a simple planar contact boundary was defined in the virtual environment and the device end-effector was manually moved toward and against the wall during repeated interaction trials.

The purpose of this test was to verify that the proxy-based rendering method correctly prevented penetration of the constrained proxy through the wall surface while allowing the unconstrained device position to continue moving, thereby generating a restoring virtual coupling force.

The test was used to assess both the computed force response of the model and its behaviour when coupled to the physical device.

The following behaviours were examined:

\begin{itemize}
	\item absence of significant rendered force when the end-effector remained outside the wall
	\item increase in rendered force as penetration depth increased
	\item qualitative agreement between the observed force response and the implemented spring--damper coupling model
\end{itemize}

\subsubsection{Constrained Motion Along a Virtual Surface}

In addition to evaluating normal contact behaviour, the system was tested for its ability to support constrained motion along a virtual surface.

During this test, the device end-effector was brought into contact with a planar virtual wall and allowed to move freely along the surface while maintaining contact. This behaviour arises naturally from the proxy-based rendering method, in which motion normal to the surface is resisted while tangential motion remains unconstrained.

The following aspects were assessed qualitatively:

\begin{itemize}
	\item ability to maintain contact with the surface while moving along it
	\item absence of significant resistance to tangential motion
	\item stability of motion during sustained sliding contact
	\item smoothness of transitions between free-space and constrained motion
\end{itemize}

This test demonstrated that the system could provide implicit motion guidance along surfaces, which is relevant for tasks involving contact-based manipulation and training scenarios.


\subsection{Safety Validation}
Safety validation tests were conducted to ensure that the system remained bounded and behaved predictably under both normal and fault conditions. These tests focused on verifying the effectiveness of software and embedded safety mechanisms, including torque saturation, rate limiting, and watchdog-based fail-safe behaviour.
\subsubsection{Torque Saturation Test}

A torque saturation test was performed to verify that actuator command limits were correctly enforced across both the host and embedded control layers.

Force outputs generated by the haptic rendering engine were limited on the host side to a predefined maximum value. In addition, the embedded controller applied a secondary torque clamp to ensure that actuator commands remained within safe operating bounds regardless of incoming data.

During testing, interaction scenarios producing large virtual forces were introduced, and the resulting actuator commands were observed to confirm that:

\begin{itemize}
	\item force commands were correctly limited on the host side
	\item transmitted torque commands were further bounded by the embedded controller
	\item actuator output remained within safe and predictable limits under all conditions
\end{itemize}

This layered saturation approach provided robustness against both software and communication-related anomalies.

\subsubsection{Rate Limiting Test}

A rate limiting test was conducted to evaluate the effect of limiting the rate of change of commanded torque at the embedded controller level.

A slew-rate limiter was implemented on the microcontroller to constrain the maximum rate at which actuator torque commands could change. This was introduced to reduce high-frequency oscillations and improve stability during rapid contact transitions.

During testing, interaction scenarios involving abrupt changes in virtual force were generated, and the resulting actuator behaviour was observed to determine whether:

\begin{itemize}
	\item rapid command changes were smoothed into bounded torque ramps
	\item high-frequency oscillations were reduced during contact events
	\item overall system stability was improved during interaction
\end{itemize}

This mechanism was introduced as a stability enhancement to mitigate oscillatory behaviour in the haptic feedback loop.

\subsubsection{Watchdog Timeout Test}

A watchdog timeout test was performed to verify safe system behaviour under loss of communication between the host and embedded controller. In this test, the stream of valid incoming torque commands was intentionally interrupted during operation.

The response of the embedded controller was then observed to confirm that actuator torque commands were reset to zero after the predefined timeout interval.

The following conditions were checked:

\begin{itemize}
	\item whether the watchdog triggered reliably when command updates ceased
	\item whether commanded torque was removed within the intended timeout window
	\item whether the device returned to a safe passive state after communication loss
\end{itemize}

This test was used to validate the fail-safe behaviour of the embedded control layer in the event of communication interruption or host-side failure.



\subsection{Hardware Validation}
Hardware validation tests were performed to assess the behaviour of the physical actuation system and the overall device during interaction. These tests focused on qualitative evaluation of actuator responsiveness and the stability of sustained user interaction with the haptic interface.
\subsubsection{Actuator Torque Response Characterisation}

Basic actuator response behaviour was assessed using a standalone motor test prior to full system integration. In this test, step changes in commanded torque were applied to an individual actuator, and the resulting mechanical response was evaluated qualitatively.

The purpose of this test was to verify that the motor and control hardware could produce responsive and stable torque output suitable for subsequent use in the haptic interface.

The evaluation focused on:

\begin{itemize}
	\item qualitative responsiveness to changes in commanded torque
	\item presence of oscillatory or unstable behaviour
	\item smoothness of the resulting motion and resistive feel
	\item repeatability across repeated command inputs
\end{itemize}

The assessment was performed through direct observation and manual interaction with the actuator, and therefore represents a qualitative validation of actuator behaviour rather than a precise quantitative characterisation.

The behaviour observed during this standalone test was consistent with the actuator performance required for stable force rendering in the integrated haptic system.

\subsubsection{Stable Contact Test}

A stable contact test was conducted to evaluate the behaviour of the full closed-loop system during sustained interaction with a virtual surface.

In this test,the device end-effector was brought into contact with a virtual object and maintained contact while observing the resulting force feedback behaviour.

Unlike the virtual wall test, which focuses primarily on validating the contact model, this test evaluates the stability and usability of the integrated system under continuous interaction conditions.

The following aspects were examined:

\begin{itemize}
	\item ability to maintain continuous contact without runaway motion
	\item presence of oscillation, chatter, or instability
	\item consistency of the rendered resistive force
	\item qualitative usability of the interaction
\end{itemize}

This test served as an integrated validation of the practical haptic stability of the prototype under representative user interaction conditions.
